# Phase 2 plan — iOS xcframework Embed & Sign for native addons

Companion to [`build-architecture-plan.md`](./build-architecture-plan.md) §5
Phase 2. The canonical Phase 2 covers BOTH platforms — Android `jniLibs/`
+ iOS xcframework + Android stdio JNI fix + a unified `__loadAddon`
runtime helper. This plan narrows the first iteration to **iOS only**;
Android packaging stays on the Phase 1 asset-extraction path. Reasons in
§0.

The Phase 1 review surfaced one concrete iOS deferral that this plan
delivers: replace the `nodejs-native/<arch>/` resource bundle (extracted
at runtime via `AppLifecycleDelegate.prepareNodeBundle()`) with one
`<name>@<version>.xcframework` per native module, embedded by Xcode's
standard Embed & Sign phase. The rolled-up backend's `process.dlopen`
loads from `Bundle.main.bundlePath/Frameworks/` directly — zero copies,
codesigned for free.

---

## 0. Scope and framing

### Why iOS-only first

The canonical Phase 2 in `build-architecture-plan.md` §5 phases iOS +
Android together. Splitting them isn't a regression of intent — it's a
risk-management call.

- Android packaging works today via the Phase 1 asset-extraction path
  (`android/src/main/assets/nodejs-native/<abi>/...`). It's stable, CI
  is green, the `Instrumented Tests (30)` job exercises the full backend
  on an emulator. There's no user-visible problem to fix on Android in
  this iteration.
- iOS Phase 1 left a known, documented architectural shortcut in place:
  the runtime asset overlay in `prepareNodeBundle()`, which the Phase 1
  review explicitly tagged as Phase 2 work. It also introduces real
  cold-start cost (~24 MB re-extracted on every launch).
- Migrating Android to `jniLibs/` requires touching:
  - `scripts/build-backend.ts` (output path and filename),
  - `android/build.gradle` (`jniLibs.srcDirs`, `packagingOptions`),
  - `android/src/main/AndroidManifest.xml` (`extractNativeLibs="false"`),
  - the `NodeJSService.kt` JNI host (stdio pump drain fix per
    `build-architecture-plan.md` §0.4),
  - the rolled-up bundle's loader resolution (one `__loadAddon` helper
    or two, depending on whether iOS migrates the loader pattern in this
    same PR).

  Each of those has the potential to break Android instrumented tests,
  and the on-emulator iteration loop is ~12 minutes. Bundling that
  diagnostic surface with iOS xcframework work makes failures harder to
  isolate.
- The iOS xcframework migration ships a clear win on its own:
  zero-extraction native loading, automatic codesigning, ~24 MB / 50
  files less I/O at every cold start. It does not block, depend on, or
  invalidate the Android `jniLibs/` migration that follows.

### Phase 2.5 / 2.6 follow-ups (out of scope here)

- **Android `jniLibs/` migration** — same architectural shape as iOS;
  parallel PR after this lands.
- **`NodeJSService.kt` JNI stdio pump drain fix** (canonical §0.4) —
  separable from packaging; pair with the `jniLibs/` PR or its own.
- **Unified `__loadAddon(name, version)` helper** (canonical §6,
  Appendix C option A) — only meaningful once Android also moves off
  the bare-addon-resolver path. This iteration keeps Android on the
  current `rollup-plugin-native-paths` rewrites and adds an iOS-only
  loader rewrite that uses `process.dlopen` against
  `NATIVE_LIB_DIR/<name>@<version>.framework/<name>@<version>`.
- **Android stdio race**, **maps re-introduction**, **fetch polyfill**,
  **smoke test on real iOS device** — tracked in
  [unified-js-bundle-ios-plan.md §7](./unified-js-bundle-ios-plan.md#7-phase-2-follow-ups).

### What ships in this branch

- iOS device + simulator support via xcframework (Phase 1 was
  simulator-only).
- `<name>@<version>.xcframework` per native module, generated by
  `scripts/build-backend.ts` from the existing per-module GitHub
  Release prebuilds.
- `ios/ComapeoCore.podspec` switches from `s.resources = ['nodejs-project', 'nodejs-native']`
  to `s.resources = ['nodejs-project']` + per-xcframework
  `s.vendored_frameworks` entries.
- `AppLifecycleDelegate.prepareNodeBundle()` collapses to a JS-only
  copy. The arch-slice merge step is gone.
- `NodeJSService.swift` exports `NATIVE_LIB_DIR` to nodejs-mobile so
  the rolled-up bundle's iOS-specific loader rewrite finds the
  embedded frameworks.
- `backend/rollup.config.js` gains an iOS-only loader rewrite plugin
  that converts the three loader patterns to
  `process.dlopen(mod, path.join(NATIVE_LIB_DIR, '<name>@<version>.framework/<name>@<version>'))`.
  Android's existing `rollup-plugin-native-paths` is untouched.
- iOS-tests CI now runs both simulator (per Phase 1) and a device-arch
  build verification (xcodebuild build, no run — codesign succeeds).

## 1. Current state (Phase 1 baseline)

```
ios/
├── ComapeoCore.podspec       # s.resources = ['nodejs-project', 'nodejs-native']
├── nodejs-project/           # rolled-up backend (gitignored, generated)
│   ├── index.mjs
│   ├── node_modules/         # JS deps + native module package.json/binding.gyp
│   └── package.json
├── nodejs-native/            # iOS prebuild .node files, gitignored
│   ├── arm64-simulator/
│   │   └── node_modules/<pkg>/prebuilds/ios-arm64/<pkg>.node   ← .node, not framework
│   └── x64-simulator/
│       └── node_modules/<pkg>/prebuilds/ios-arm64/<pkg>.node
└── NodeMobile.xcframework    # embedded Node runtime; vendored_frameworks
```

At launch, `AppLifecycleDelegate.prepareNodeBundle()` copies
`nodejs-project/` into Application Support and overlays the active
`nodejs-native/<arch>/` slice on top, producing
`Library/Application Support/comapeo-runtime/nodejs-project/`. Node
starts with that directory's `index.mjs` as its entry. The bare addon
resolvers (`bindings`, `node-gyp-build`, `require.addon`) — patched at
build time by `backend/rollup-plugins/rollup-plugin-native-paths.js` —
resolve to `node_modules/<pkg>/prebuilds/ios-<arch>/<pkg>.node` and
`process.dlopen` succeeds.

Two limitations, both deliberate Phase 1 scopes:

- **Cold-start cost.** ~50-file / ~24 MB re-extraction on every launch.
- **Simulator-only.** The xcframework migration is the prerequisite for
  device support; a one-arch device hack would be Phase-2-throwaway
  work.

## 2. Target state

```
ios/
├── ComapeoCore.podspec       # s.resources = ['nodejs-project']
│                             # s.vendored_frameworks = ['NodeMobile.xcframework',
│                             #                          'Frameworks/*.xcframework']
├── nodejs-project/           # rolled-up backend (unchanged on disk)
├── Frameworks/               # NEW — gitignored, generated by build-backend.ts
│   ├── better-sqlite3@11.10.0.xcframework
│   ├── crc-native@1.1.8.xcframework
│   ├── fs-native-extensions@1.5.0.xcframework
│   ├── quickbit-native@2.4.8.xcframework
│   ├── rabin-native@2.0.0.xcframework
│   ├── simdle-native@1.3.9.xcframework
│   └── sodium-native@5.1.0.xcframework
└── NodeMobile.xcframework    # unchanged
```

Each `<name>@<version>.xcframework` contains both:

- `ios-arm64/<name>@<version>.framework/<name>@<version>` — device
- `ios-arm64_x86_64-simulator/<name>@<version>.framework/<name>@<version>`
  — fat simulator (lipo of arm64-simulator + x64-simulator)

Mach-O install name on each binary:
`@rpath/<name>@<version>.framework/<name>@<version>`. Apple's
`xcodebuild -create-xcframework` produces the wrapper.

At app build time, Xcode's Embed & Sign phase places the right slice
under `<App>.app/Frameworks/<name>@<version>.framework/` and codesigns
it. At runtime:

```js
process.dlopen(
  mod,
  path.join(
    NATIVE_LIB_DIR,                        // = Bundle.main.bundlePath + "/Frameworks"
    `${name}@${version}.framework/${name}@${version}`,
  ),
)
```

The runtime helper is injected by an iOS-only rollup plugin alongside
the existing `rollup-plugin-native-paths` (which still runs for the
Android output). On Android nothing changes.

`AppLifecycleDelegate.prepareNodeBundle()` becomes:

```swift
private static func prepareNodeBundle() -> String? {
    // Copy `nodejs-project/` → Application Support so nodejs-mobile
    // gets a writable filesystem path for the entry. No native overlay
    // — the .node binaries live in <App>.app/Frameworks/ (iOS Embed &
    // Sign phase) and load via process.dlopen at runtime.
}
```

## 3. Implementation steps

### Step 1 — Extend `scripts/build-backend.ts` to emit xcframeworks

Add a fourth pass after the existing prebuild fetch. Recipe lifted
from the validated harness in
`digidem/nodejs-mobile-bare-prebuilds@feat/jnilibs-xcframework-packaging`
(`assemble-test-project/action.yml#Wrap addons as xcframeworks`):

1. For each native module:
   1. For each iOS arch slice (`ios-arm64`, `ios-arm64-simulator`,
      `ios-x64-simulator`): take the `.node` file, copy it as the
      Mach-O exec inside a `<name>.framework/` directory, and synthesise
      a minimal `Info.plist` next to it.
   2. `lipo -create` the two simulator binaries into one fat Mach-O,
      producing a single simulator `<name>.framework/<name>` binary
      that covers both Apple Silicon and Intel hosts.
   3. `xcodebuild -create-xcframework -framework <device-fwk> -framework <sim-fat-fwk> -output ios/Frameworks/<name>.xcframework`.
2. The existing `ios/nodejs-native/<arch>/` write-out is **deleted**.
   Phase 1's output trees should be removed cleanly so a downstream
   developer who merges main and reinstalls doesn't see stale
   directories.
3. `IOS_ARCHS = ["arm64", "arm64-simulator", "x64-simulator"] as const`
   — adds the device slice.

**Skipped for now (matches harness):**

- **No versioned framework names** (`<name>.framework`, not
  `<name>@<version>.framework`). The 7 native modules ship as a single
  version each in our backend's dep tree today; multi-version handling
  is a future concern and the canonical plan explicitly punts it to
  "Phase 2.5 / 2.6". Adding the version suffix later is mechanical
  (rollup rewrite + filename string) once a multi-version need appears.
- **No `install_name_tool -id` step.** The Mach-O install name only
  matters when something else does an rpath-based lookup; we always
  hand `process.dlopen` a full path
  (`NATIVE_LIB_DIR/<name>.framework/<name>`), so the embedded install
  name is irrelevant. The harness ships without it.

`xcodebuild -create-xcframework` requires Xcode tooling, so this only
runs on macOS. The Android workflow doesn't currently run
`scripts/build-backend.ts` (it has its own download path), so no CI
churn there.

### Step 2 — `ios/ComapeoCore.podspec`

```ruby
s.resources = ['nodejs-project']
s.vendored_frameworks = [
  'NodeMobile.xcframework',
  'Frameworks/*.xcframework',
]
```

CocoaPods globs `vendored_frameworks` and emits Embed-&-Sign phases for
each; no manual app-side config needed. The existing
`s.static_framework = true` stays — it controls the ComapeoCore pod's
own framework, not the embedded Node modules.

### Step 3 — `ios/NodeJSService.swift`: pass `NATIVE_LIB_DIR`

Two surface changes to `runNode()`:

- Compute `nativeLibDir = Bundle.main.bundlePath + "/Frameworks"`.
- Pass it as an environment variable (not argv) so the rolled-up loader
  can read it via `process.env.NATIVE_LIB_DIR` without conflicting with
  the existing argv shape (`[node, --no-experimental-fetch, jsPath,
  comapeoSocketPath, controlSocketPath, privateStorageDir]`).
  `nodejs-mobile`'s `NodeMobileStartNode` inherits `setenv()` values
  from the calling thread.

### Step 4 — Patch `@comapeo/core` to accept a pre-loaded better-sqlite3 binding

better-sqlite3 differs from the holepunch-style addons in two ways
that affect the rollup rewrite:

- **Lazy load.** `@comapeo/core` does
  `import Database from 'better-sqlite3'` at module-init, but the
  `require('bindings')('better_sqlite3.node')` call inside
  `database.js#L48` only fires the first time `new Database(...)` runs.
- **Underscore-vs-hyphen.** The .node binary is named
  `better_sqlite3.node` (underscore), not the package name with hyphen.
  Our xcframework would land at `better-sqlite3.framework/better-sqlite3`
  (hyphen, matching the npm package name); the original
  `require('bindings')('better_sqlite3.node')` lookup has no chance of
  reaching it.

Fortunately, `better-sqlite3@11.10.0` already has built-in support for
externally-loaded addons via the `nativeBinding` option
(`database.js#L36-56`): pass the option an addon module object, and the
internal `require('bindings')` call is bypassed entirely. We drive that
route by patching `@comapeo/core` to accept a new option,
**`betterSqlite3NativeBinding`** (mirrors better-sqlite3's
`nativeBinding` naming so an upstream-docs reader recognises it
instantly), and threading it to both `new Database(...)` callsites
(`mapeo-manager.js#L204`, `mapeo-project.js#L211`).

`backend/lib/create-comapeo.js` populates the new option with
`__loadAddon('better-sqlite3')` (defined in step 5). Holepunch-style
modules don't take the same path because their loader call patterns
(`bindings`, `node-gyp-build`, `require.addon`) fire eagerly at
module-init; the rollup rewrite in step 5 catches those at the bundle
level.

Patch mechanism options, in increasing order of footprint:

- **Patch-package** patch checked into `backend/patches/` — visible in
  source control, runs automatically on `npm install`, the de-facto
  industry pattern.
- **Rollup-time alias** to a small in-repo replacement file (similar to
  the Phase 1 `backend/lib/maps-stub.js` pattern). Ergonomic for our
  build but invisible to anyone running `npm install` outside the build
  pipeline.
- **Upstream PR**. Best long-term home; not blocking for Phase 2.

Recommend **patch-package** for this iteration: the patch survives
upstream `@comapeo/core` releases unchanged when the affected
constructor signatures don't move, and it doesn't re-introduce the
"node_modules mutation between npm ci and rollup" concern Phase 1's
review flagged.

### Step 5 — iOS rollup loader rewrite

New plugin in `backend/rollup-plugins/rollup-plugin-ios-addon-loader.js`,
applied only to the iOS output of `backend/rollup.config.js`. It
rewrites the same three loader patterns
`rollup-plugin-native-paths.js` matches, but to a single
`__loadAddon(name)` call. Also exports `__loadAddon` so
`backend/lib/create-comapeo.js` can pass the pre-loaded binding into
`MapeoManager`'s new `betterSqlite3NativeBinding` option (step 4).

```js
// Injected via output.banner for the iOS bundle only.
const __nodeRequire = require
const __nativeLibDir = process.env.NATIVE_LIB_DIR
const __addonCache = new Map()

function __loadAddon(name) {
  const cached = __addonCache.get(name)
  if (cached) return cached
  const mod = { exports: {} }
  process.dlopen(
    mod,
    __nodeRequire('path').join(__nativeLibDir, name + '.framework', name),
  )
  __addonCache.set(name, mod.exports)
  return mod.exports
}

globalThis.__loadAddon = __loadAddon
```

The package name at each rewritten callsite comes from the file's
`package.json` walk at rollup transform time — same source of truth as
the xcframework filename in step 1.

`backend/rollup.config.js` now applies different addon-resolver plugins
per output:

- Android output: `rollup-plugin-native-paths` (unchanged).
- iOS output: `rollup-plugin-ios-addon-loader` (new) + maps-plugin
  alias (Phase 1, unchanged).

### Step 6 — `AppLifecycleDelegate.prepareNodeBundle()` simplification

Drop the `mergeDirectory()` step. `prepareNodeBundle()` now just copies
`Bundle.main.bundlePath/nodejs-project/` to
`Library/Application Support/comapeo-runtime/nodejs-project/` and
returns the merged `index.mjs` path. The arch-slice picker
(`#if arch(arm64)` block) is removed entirely — no per-arch branch
exists in the resource tree any more.

`mergeDirectory()` itself is dead code after this and should be
deleted. Keep `prepareNodeBundle()` because nodejs-mobile still wants a
filesystem path for the entry, and the rolled-up bundle ships read-only
inside the `.app` bundle.

### Step 7 — Tests

- Existing `CoreManagerSmokeTest` runs unchanged against the simulator;
  asserts `ready` over the control socket. Should pass without
  modification because the only thing that changes from the bundle's
  perspective is *where* `process.dlopen` resolves the .node binary
  from.
- New `IOSDeviceArchBuildTest`: a CI job that runs
  `xcodebuild build -sdk iphoneos -destination 'generic/platform=iOS'
  CODE_SIGNING_ALLOWED=NO` to confirm the device slice links and
  codesigns. No run — Apple Silicon CI runners can't simulate device
  hardware. The build alone proves the xcframework's device slice is
  valid.
- `ServiceLifecycleTest` should continue passing; nothing about the
  service lifecycle changes.

## 4. Risks

1. **`install_name_tool -id` on prebuilt `.node` files.** The .node
   files we receive from the per-module GitHub Releases were built as
   loadable bundles, not frameworks. Their existing install names may
   be `@rpath/<some-name>.node` or unset. Setting a new install name is
   a one-shot operation that's well-supported, but downstream
   `process.dlopen` only works if Apple's loader can match the install
   name against the embedded path. Validate by inspecting the output
   binary with `otool -l | grep LC_ID_DYLIB` after each step-1 wrap.

2. **xcframework codesigning of unsigned `.node` Mach-Os.** The .node
   binaries from per-module releases are unsigned. Embed & Sign should
   handle this — Xcode applies a codesign step to anything in
   `Frameworks/` regardless of prior state. If it doesn't, fallback is
   a one-line `codesign --remove-signature` in step 1 to be sure the
   start state is clean. Phase 2 testing is the time to find out.

3. **`Info.plist` incompatibility.** Apple expects `CFBundleExecutable`,
   `CFBundleIdentifier`, `MinimumOSVersion`, etc. The plist template
   inside `<name>@<version>.framework/Info.plist` needs to match what
   the loader and codesign tool expect. Crib the one
   `digidem/nodejs-mobile-bare-prebuilds`'s test harness uses
   (validated end-to-end per `build-architecture-plan.md` §0).

4. **`@` in framework names.** `<name>@<version>.framework` puts an `@`
   in the path. Apple's `xcodebuild`, `install_name_tool`, and
   `codesign` all accept it (validated by harness, per canonical §0
   point 2). If `pod install` rejects `@` in a `vendored_frameworks`
   glob match, the fallback is `<name>__<version>.framework` or just
   `<name>.framework` (single-version assumption). The harness uses
   `@`; trust it.

5. **`@rpath` resolution at app launch.** The Mach-O install name is
   `@rpath/<name>@<version>.framework/<name>@<version>`. The host
   binary has an `LC_RPATH` of `@executable_path/Frameworks` by
   default; Embed & Sign ensures the framework lands there. If Xcode's
   embed phase places the framework somewhere else (e.g. nested), the
   loader will fail with "library not loaded". `otool -l <App>.app/<App> | grep LC_RPATH`
   after pod install + first build to confirm the rpath. This is
   well-trodden territory but worth the explicit check.

6. **Static-framework conflict.** `s.static_framework = true` in
   `ComapeoCore.podspec` makes the *ComapeoCore* pod itself a static
   framework. The vendored xcframeworks ship as dynamic frameworks
   (Mach-O bundles loaded at runtime via `dlopen`). Mixing dynamic and
   static frameworks via CocoaPods is supported; just confirm
   Embed & Sign actually fires for the vendored entries with
   `static_framework = true` set. If Embed-only-without-Sign happens,
   binaries may load unsigned at runtime and trip the iOS code-signing
   enforcement on real devices (simulator is more permissive).

7. **CI device-build job is new.** macos-15 runners can run
   `xcodebuild build -sdk iphoneos` as long as we don't ask for a
   device destination. Build-only — exit succeeds when codesign
   succeeds. Time cost is roughly the same as the existing simulator
   build (no testing).

8. **`NATIVE_LIB_DIR` env var inheritance through nodejs-mobile.**
   Setenv'd values inherit through `NodeMobileStartNode` because it
   spawns Node in the same process. The bundle reads `process.env`
   inside the V8 context. Confirmed pattern from canonical plan §4
   "Proposed state". Validate by adding a `console.log` in
   `__loadAddon` that prints `process.env.NATIVE_LIB_DIR` once on
   first invocation.

## 5. Acceptance criteria

- [ ] `npm run backend:build` produces `ios/Frameworks/<name>@<version>.xcframework`
      for each of the 7 native modules.
- [ ] `git ls-files ios/Frameworks/` returns nothing (gitignored).
- [ ] `git ls-files ios/nodejs-native/` returns nothing AND the
      directory is no longer produced (build-backend.ts emits no

- [ ] `cd example && npx expo prebuild --platform ios && cd ios && pod install`
      succeeds with the new podspec; `<App>.app/Frameworks/` contains
      one `<name>@<version>.framework/` per native module after
      `xcodebuild build`.
- [ ] `otool -l <App>.app/Frameworks/sodium-native@5.1.0.framework/sodium-native@5.1.0 | grep LC_ID_DYLIB`
      prints
      `name @rpath/sodium-native@5.1.0.framework/sodium-native@5.1.0`.
- [ ] `CoreManagerSmokeTest` passes on iOS simulator.
- [ ] `ServiceLifecycleTest` passes on iOS simulator.
- [ ] iOS device build job (new CI matrix axis) succeeds; codesign
      succeeds for all xcframework slices.
- [ ] `AppLifecycleDelegate.prepareNodeBundle()` no longer references
      `nodejs-native` and `mergeDirectory()` is deleted.
- [ ] Android instrumented tests still pass — confirms iOS-only scope
      did not regress the platform we're not touching.

## 6. Commit order

Recommend splitting the branch into ~4 commits so the diff is
reviewable:

1. **`scripts/build-backend.ts`: emit xcframeworks.** The biggest pure
   build-pipeline change. Removes the old `ios/nodejs-native/` write
   path. After this commit, `npm run backend:build` produces the
   target tree.
2. **`backend/rollup.config.js` + new `rollup-plugin-ios-addon-loader.js`.**
   The iOS-only loader rewrite. Verified by inspecting the diff between
   `dist/ios/index.mjs` and `dist/android/index.mjs`.
3. **`ios/ComapeoCore.podspec` + `ios/NodeJSService.swift` + `ios/AppLifecycleDelegate.swift`.**
   Wiring: vendored frameworks, `NATIVE_LIB_DIR` env var, dropping
   `mergeDirectory()`. After this, the simulator smoke test should
   pass.
4. **`.github/workflows/ios-tests.yml`: device-arch build job.** Adds
   the codesign-only verification on `iphoneos`. Last because it's the
   only step that can be diagnosed independently of the rest landing.

A fifth commit may be needed for `.gitignore` cleanup
(`ios/Frameworks/` added; `ios/nodejs-native/` retained as a guard
rail).

## 7. Defer / out of scope

- Android `jniLibs/` migration. Tracked by canonical Phase 2; will be
  its own branch.
- `NodeJSService.kt` JNI stdio fix. Same.
- Unified `__loadAddon` helper across both platforms. Becomes
  meaningful once Android also moves off `rollup-plugin-native-paths`.
- Real-device smoke test (an XCTest that runs on a device, not just a
  build). Phase 2.5.
- Version-stamp gate on `prepareNodeBundle()`. Phase 2.5; gated on the
  Phase 2 simplification of that function landing first so the
  optimisation has a smaller surface to gate.
- iOS map-tile fetching, fetch polyfill, maps-stub `console.warn`. All
  unrelated to packaging; tracked in
  [unified-js-bundle-ios-plan.md §7](./unified-js-bundle-ios-plan.md#7-phase-2-follow-ups).
